INTRO DDDの全体像

ドメイン駆動設計（DDD）は、Eric Evans が2003年に提唱した「ソフトウェアの核心にある複雑さに立ち向かう」ための設計手法です。技術ではなくビジネスドメインの理解を設計の中心に据えます。

DDDの根本思想: ソフトウェアの構造は、ビジネスドメインの構造を反映すべきである。コードはドメインエキスパートと開発者の共通言語で書かれるべきである。

戦略的設計 vs 戦術的設計

DDDは大きく2つのレベルに分かれます。クリックして詳細を確認してください。

チーム全員が使う共通のドメイン用語を定義し、コードにも反映する

戦略的設計

戦術的設計

戦術的設計

データの整合性境界と永続化の抽象化パターン

戦術的設計

戦術的設計

よくある誤解: DDDは「戦術的パターン（Entity, VO, Repository…）を使うこと」ではありません。本質は戦略的設計 — ドメインの深い理解とモデリングにあります。パターンは手段に過ぎません。

戦略 ユビキタス言語

ユビキタス言語（Ubiquitous Language）は DDDの最も重要な概念です。ドメインエキスパートと開発者が同じ言葉を使い、その言葉がそのままコードのクラス名・メソッド名になります。

なぜ重要？ 「仕様書」と「コード」で別の言葉を使うと、翻訳コストが発生し、誤解・バグの温床になる。言葉を統一すれば、会話がそのまま設計になる。

悪い例 → 良い例

ECサイトのドメインで考えてみましょう。用語をクリックすると解説が表示されます。

開発者用語（技術寄り）

user_data_record → Customer（顧客）
「データレコード」は技術用語。ビジネスでは「顧客」と呼ぶ。クラス名も Customer に。
item_list → Cart（カート）
「アイテムリスト」はデータ構造の名前。「カートに入れる」がドメインの言葉。
flag_is_done → Order.fulfill()（注文を履行する）
フラグの ON/OFF ではなく「注文を履行する」というドメインの振る舞いとして表現する。

ドメイン言語で書くコード

// ❌ 技術用語ベース
function updateStatus(record, flag) {
  record.status_flag = flag;
}

// ✅ ユビキタス言語ベース
class Order {
  fulfill() {
    if (!this.canFulfill()) {
      throw new Error("在庫不足で履行不可");
    }
    this.status = OrderStatus.Fulfilled;
    this.fulfilledAt = new Date();
  }
}

実践のポイント

  • 用語集を作る — プロジェクト Wiki やコードの README に「この文脈での用語」を定義する
  • コードレビューで言葉を確認 — 変数名・メソッド名がドメイン用語と一致しているか？
  • ドメインエキスパートとの会話をそのまま設計に — 「注文をキャンセルする」なら order.cancel()

戦略 境界づけられたコンテキスト

Bounded Context（境界づけられたコンテキスト）は、モデルが一貫した意味を持つ範囲を明確に区切るものです。同じ「商品」でも、販売・在庫・配送では意味が異なります。

核心: 1つの巨大なモデルを作るのではなく、文脈ごとにモデルを分ける。各コンテキスト内ではユビキタス言語が統一される。

同じ言葉、違う意味

「商品（Product）」がコンテキストによってどう変わるか見てみましょう。ボックスをクリックして詳細を確認できます。

販売コンテキスト
Product

名前、価格、説明文、カテゴリ、レビュー評価

class Product {
  name: string
  price: Money
  description: string
  category: Category
  applyDiscount(rate) { ... }
}
在庫コンテキスト
Product

SKU、在庫数、倉庫位置、入出庫履歴

class StockItem {
  sku: SKU
  quantity: number
  warehouse: Location
  reserve(qty) { ... }
  restock(qty) { ... }
}
配送コンテキスト
Product

重量、サイズ、配送区分、取り扱い注意フラグ

class Parcel {
  weight: Weight
  dimensions: Dimensions
  shippingClass: ShippingClass
  calculateShippingCost() { ... }
}

コンテキストマップ

コンテキスト間の関係を「コンテキストマップ」で表現します。代表的なパターン:

  • 共有カーネル (Shared Kernel) — 2つのコンテキストがモデルの一部を共有。変更には両チームの合意が必要
  • 顧客/供給者 (Customer/Supplier) — 上流が下流のニーズに合わせてAPIを提供
  • 腐敗防止層 (Anti-Corruption Layer) — 外部の汚いモデルを自分のドメインモデルに変換するアダプター層
  • 公開ホストサービス (Open Host Service) — 公開APIを通じて複数の下流コンテキストにサービス提供
実践のコツ: マイクロサービスに分割する前に、まずコンテキスト境界を見つけよう。境界を間違えたままサービス分割すると、分散モノリスという最悪の結果になる。

戦術 エンティティ vs 値オブジェクト

ドメインの概念を「誰であるかが重要か（=エンティティ）」と「何であるかが重要か（=値オブジェクト）」に分類するのが、DDDの戦術的設計の出発点です。

エンティティ (Entity)

同一性（Identity）で区別
  • 一意のIDを持つ
  • ライフサイクルがある（生成→変更→削除）
  • 属性が変わっても「同じもの」
class Customer {
  readonly id: CustomerId  // 同一性
  name: string              // 変わりうる
  email: Email             // 変わりうる

  changeName(newName) {
    this.name = newName;    // IDは変わらない
  }
}
// 名前が変わっても同じ顧客
例: 顧客、注文、商品、ユーザーアカウント

値オブジェクト (Value Object)

属性の組み合わせで区別
  • IDを持たない
  • 不変（Immutable）
  • 属性が全て同じなら「同じもの」
class Money {
  readonly amount: number
  readonly currency: string

  add(other: Money): Money {
    // 新しいインスタンスを返す
    return new Money(
      this.amount + other.amount,
      this.currency
    );
  }
}
// 100円 === 100円（中身で比較）
例: 金額、住所、メールアドレス、期間、色

判定フローチャート

そのドメイン概念には「一意のID」で追跡する必要がある？
属性が全て同じなら「同じもの」と見なせる？
→ エンティティとして設計しよう！ IDで識別し、ライフサイクルを管理する。
→ 値オブジェクトとして設計しよう！ 不変にし、等価性は属性で判定する。
設計のヒント: 迷ったら値オブジェクトを優先しよう。不変なオブジェクトはバグが少なく、テストしやすく、並行処理にも強い。エンティティは本当にIDが必要な場合だけ使う。

戦術 集約（Aggregate）

集約はデータ変更の整合性境界です。関連するエンティティと値オブジェクトをまとめ、1つのルートエンティティ（集約ルート）を通じてのみアクセスを許可します。

集約ルール: 外部から集約内部のオブジェクトを直接操作してはいけない。必ず集約ルートのメソッドを通す。1トランザクション = 1集約の更新。

注文（Order）集約の例

集約ルート
Order
Entity OrderLine
注文明細。商品ID・数量・単価を持つ。Orderを通じてのみ追加・削除される。
Value Object ShippingAddress
配送先住所。不変。変更時は新しい値オブジェクトで置き換え。
Value Object Money (totalAmount)
合計金額。OrderLineから自動計算。直接設定は不可。
class Order {                          // 集約ルート
  readonly id: OrderId
  lines: OrderLine[] = []               // 内部エンティティ
  shippingAddress: ShippingAddress       // 値オブジェクト
  status: OrderStatus

  // ✅ 集約ルートのメソッドを通じてのみ変更
  addItem(productId, quantity, unitPrice) {
    if (this.status !== OrderStatus.Draft)
      throw new Error("確定済み注文には追加不可");
    this.lines.push(new OrderLine(productId, quantity, unitPrice));
  }

  get totalAmount(): Money {            // 整合性を保証
    return this.lines.reduce(
      (sum, line) => sum.add(line.subtotal), Money.zero("JPY")
    );
  }

  confirm() {
    if (this.lines.length === 0)
      throw new Error("空の注文は確定不可");
    this.status = OrderStatus.Confirmed;
  }
}

// ❌ NG: 外部から内部を直接操作
order.lines.push(newLine);           // 整合性チェックをバイパス！
order.lines[0].quantity = 999;       // 合計金額と不整合！

// ✅ OK: 集約ルートを通す
order.addItem(productId, 2, price);  // バリデーション済み

集約設計の原則

  • 小さく保つ — 集約が大きいとロック競合が増える。必要最小限のエンティティだけ含める
  • IDで参照 — 他の集約を直接保持せず、IDで参照する（Order は CustomerId を持つが Customer オブジェクトは持たない）
  • 結果整合性を受け入れる — 集約間の整合性はドメインイベントで非同期に保つ

戦術 ドメインイベント

ドメインイベントは「ドメインで起きた重要な出来事」を表現するオブジェクトです。過去形で命名し、不変です。集約間・コンテキスト間の連携を疎結合に実現します。

命名規則: 「〜された」という過去形で。OrderConfirmed, PaymentReceived, ItemShipped。動詞の命令形（ConfirmOrder）はコマンドであり、イベントではない。

イベントフロー体験

「注文確定」ボタンを押して、ドメインイベントがどう連鎖するか見てみましょう。

OrderConfirmed
注文集約が確定状態に。イベントを発行。
→ PaymentRequested
決済コンテキストが受信。支払い処理を開始。
→ InventoryReserved
在庫コンテキストが受信。在庫を引き当て。
→ NotificationSent
通知コンテキストが受信。確認メールを送信。
// イベントの定義（不変）
class OrderConfirmed {
  readonly orderId: OrderId
  readonly totalAmount: Money
  readonly occurredAt: Date
}

// 集約内でイベントを発行
class Order {
  confirm() {
    this.status = OrderStatus.Confirmed;
    this.addDomainEvent(new OrderConfirmed(
      this.id, this.totalAmount, new Date()
    ));
  }
}

// 別のコンテキストのハンドラーが反応
class PaymentHandler {
  handle(event: OrderConfirmed) {
    paymentService.requestPayment(event.orderId, event.totalAmount);
  }
}

戦術 リポジトリパターン

リポジトリは集約の永続化を抽象化するパターンです。ドメイン層はデータベースの存在を知らず、「コレクション」のようなインターフェースを通じて集約を取得・保存します。

重要: リポジトリは集約ルートごとに1つ。OrderLineRepository は作らない — OrderRepository が Order 集約全体を丸ごと保存・復元する。 
// ドメイン層: インターフェース（抽象）を定義
interface OrderRepository {
  findById(id: OrderId): Order | null
  save(order: Order): void
  findByCustomer(customerId: CustomerId): Order[]
}

// インフラ層: 具体的な実装
class PostgresOrderRepository implements OrderRepository {
  findById(id: OrderId): Order | null {
    const row = this.db.query(
      `SELECT * FROM orders WHERE id = $1`, [id.value]
    );
    return row ? this.toOrder(row) : null;
  }

  save(order: Order): void {
    this.db.transaction(() => {
      this.upsertOrder(order);
      this.upsertOrderLines(order.id, order.lines);
    });
  }
}

// アプリケーション層での使用
class ConfirmOrderUseCase {
  constructor(private orderRepo: OrderRepository) {}

  execute(orderId: string) {
    const order = this.orderRepo.findById(new OrderId(orderId));
    if (!order) throw new Error("注文が見つかりません");
    order.confirm();                    // ドメインロジック
    this.orderRepo.save(order);         // 永続化
  }
}

リポジトリ vs DAO

  • リポジトリ: 集約単位で操作。ドメインオブジェクトを返す。ドメイン層のインターフェース
  • DAO: テーブル単位で操作。行データを返す。インフラ層の概念
アンチパターン: リポジトリに findByNameAndStatusAndDateRange(...) のような複雑なクエリを増やし続けると、リポジトリが肥大化する。参照用途には CQRS（コマンドクエリ責務分離）の導入を検討しよう。

設計 レイヤードアーキテクチャ

DDDでは依存関係の方向を制御するため、アプリケーションを層に分けます。ドメイン層が中心にあり、外側の層に依存しないのが鉄則です。

プレゼンテーション層 (UI) ▼

ユーザーからの入力を受け取り、結果を表示する。REST API コントローラー、画面、CLI など。

OrderController, OrderListView
アプリケーション層 (UseCase) ▼

ユースケースを調整する薄い層。ドメインオブジェクトを取得し、ドメインのメソッドを呼び、保存する。ビジネスロジックは書かない。

ConfirmOrderUseCase, CreateOrderUseCase
ドメイン層 (核心) ▼

ビジネスルールの全て。エンティティ、値オブジェクト、集約、ドメインサービス、ドメインイベント、リポジトリのインターフェースがここに住む。

Order, Money, OrderRepository(interface), OrderConfirmed
インフラ層 (実装詳細) ▼

DB、メール、外部API、メッセージキューなどの技術的実装。ドメイン層のインターフェースを実装する。

PostgresOrderRepository, SmtpNotificationService
依存性逆転の原則 (DIP): ドメイン層がインターフェースを定義し、インフラ層がそれを実装する。これにより、ドメインロジックはDBやフレームワークに一切依存しない。テストも容易になる。 
// 依存の方向: UI → App → Domain ← Infra
//                                ↑ 逆転!

// Domain層 (インターフェースだけ定義)
interface OrderRepository { save(order): void }

// Infra層 (実装する)
class PostgresOrderRepo implements OrderRepository { ... }

// App層 (依存注入で繋ぐ)
class ConfirmOrderUseCase {
  constructor(repo: OrderRepository) {}  // 抽象に依存
}

まとめ: DDDを始めるための3ステップ

1
ドメインエキスパートと話す

技術の話はしない。ビジネスの言葉を集め、ユビキタス言語を作る。

2
境界を見つける

同じ言葉が違う意味で使われる場所が、コンテキスト境界のヒント。

3
小さく始める

最も複雑なサブドメイン1つにDDDを適用。全体に広げる前に効果を検証。
